# อธิบายระบบโค้ดภาพรวม

เอกสารนี้ใช้เป็นแนวทางอธิบายโครงงานแบบแยกเป็นส่วน ๆ โดยเน้นให้เข้าใจง่าย ใช้ภาษาคน และเน้นอธิบาย `Controller` มากกว่า `ViewModel` กับ `View` เพราะ Controller เป็นส่วนที่แสดงการทำงานของระบบได้ชัดที่สุด

## แนวทางการอธิบายแต่ละหัวข้อ

ในแต่ละระบบให้ใช้โครงแบบนี้

1. ชื่อหน้า / ชื่อระบบ
2. ViewModel ที่ใช้
3. Controller
4. View
5. ผลลัพธ์ที่ผู้ใช้เห็น
6. ตารางฐานข้อมูลที่เกี่ยวข้อง

แนวการอธิบายที่แนะนำ

- `ViewModel` อธิบายสั้น ๆ ว่าใช้ตัวไหน และใช้ property อะไรเป็นหลัก
- `View` อธิบายสั้น ๆ ว่าฟอร์มหรือหน้าจอ bind กับอะไร และส่งไป action ไหน
- `Controller` อธิบายละเอียดที่สุด เพราะเป็นส่วนตรวจสอบข้อมูล ประมวลผล ติดต่อฐานข้อมูล และกำหนดผลลัพธ์ของระบบ
- `ผลลัพธ์ที่ผู้ใช้เห็น` ให้สรุปว่าผู้ใช้เห็นอะไรเมื่อทำรายการสำเร็จหรือไม่สำเร็จ
- `ตารางฐานข้อมูลที่เกี่ยวข้อง` ให้ระบุเฉพาะตารางที่มีบทบาทจริง

รูปแบบการแคปโค้ดที่แนะนำ

- `ViewModel` แคป 1 รูปพอ ถ้าไฟล์ไม่ยาว
- `View` แคปเฉพาะส่วนฟอร์มหรือส่วนแสดงผลที่เกี่ยวข้อง
- `Controller` ให้แคปตาม `ฟังก์ชัน` เป็นหลัก
- ถ้า 1 ฟังก์ชันยาวเกิน 1 รูป ให้แคปหลายรูปต่อเนื่องกัน
- หลังจากแปะรูปในส่วน Controller แล้ว ให้ `อธิบายรวมทีเดียว` ตามลำดับการทำงานของฟังก์ชัน
- ไม่จำเป็นต้องเขียนแยกว่ารูป 1 อธิบายอะไร รูป 2 อธิบายอะไร ถ้าเป็นฟังก์ชันเดียวกัน

ตัวอย่างแนวคิด

- `1 public IActionResult` หรือ `1 public method` สามารถมองเป็น `1 ฟังก์ชัน`
- ถ้าหน้านั้นมี `GET` และ `POST` ก็ให้แคปทั้ง 2 ฟังก์ชันในหัวข้อ Controller เดียวกัน
- ถ้าฟังก์ชันใดยาวมาก เช่นมี validation, query, save, redirect ครบในตัวเดียว ให้แคป 2-3 รูป แล้วอธิบายรวมเป็นข้อ ๆ ตาม flow

## ลำดับระบบที่แนะนำให้เริ่มอธิบาย

แนะนำให้อธิบายตามลำดับการใช้งานของผู้ใช้จริง จะทำให้ภาพรวมของระบบเข้าใจง่ายที่สุด

1. สมัครสมาชิก `Register / Signup`
2. เข้าสู่ระบบ `Login`
3. กรณีลืมรหัสผ่าน `Forgot Password + OTP`
4. จัดการข้อมูลส่วนตัว `Profile + Address`
5. หน้าแรกของระบบ `HomePage`
6. ดูสินค้าและโปรโมชัน `Menu / Promotion`
7. เพิ่มสินค้าเข้าตะกร้า `AddToCart`
8. จัดการตะกร้า `Cart / Update / Remove / Clear`
9. สั่งซื้อ `Checkout`
10. ติดตามคำสั่งซื้อ `MyOrders / OrderHistory`
11. ยกเลิกคำสั่งซื้อ / ยืนยันรับสินค้า / แนบสลิป
12. ส่วนหลังบ้าน `Dashboard / OrderMenu / Product / Account / Promotion`

## รูปแบบการอธิบายที่แนะนำ 2 แบบ

### แบบที่ 1 ภาพรวมของระบบ

ใช้เมื่อต้องการอธิบายโครงสร้างการทำงานของระบบแบบเข้าใจง่าย เห็นภาพรวมก่อนลงรายละเอียด โดยเน้นเรียงตามลำดับการใช้งานจริงของผู้ใช้ตั้งแต่เริ่มต้นจนจบกระบวนการ

หัวข้อที่เหมาะสำหรับแบบภาพรวม

1. ระบบสมาชิก
2. ระบบกู้คืนรหัสผ่าน
3. ระบบแสดงข้อมูลหน้าเว็บไซต์
4. ระบบจัดการโปรไฟล์และที่อยู่
5. ระบบสั่งซื้อสินค้า
6. ระบบชำระเงินและอัปโหลดสลิป
7. ระบบติดตามคำสั่งซื้อ
8. ระบบการแจ้งเตือน
9. ระบบจัดการออเดอร์ของพนักงาน
10. ระบบจัดการสต็อกของพนักงาน
11. ระบบจัดการสมาชิกของผู้ดูแล
12. ระบบจัดการสินค้าและหมวดหมู่ของผู้ดูแล
13. ระบบจัดการโปรโมชันของผู้ดูแล
14. ระบบแดชบอร์ดผู้ดูแล

### แบบที่ 2 ละเอียดตามระบบทั้งหมด

ใช้เมื่อต้องการอธิบายให้ครบตามฟังก์ชันจริงของระบบ โดยแยกตาม action หรือฟังก์ชันหลักที่มีอยู่จริงใน Controller ของฝั่ง User, Staff และ Admin

หัวข้อที่เหมาะสำหรับแบบละเอียด

#### ฝั่งผู้ใช้ทั่วไป (User / Customer)

1. `ระบบสมัครสมาชิก : (Signup)`
   ใช้สำหรับสร้างบัญชีผู้ใช้ใหม่ โดยรับข้อมูลจากฟอร์ม ตรวจสอบข้อมูลซ้ำ และบันทึกผู้ใช้ลงฐานข้อมูล
2. `ระบบเข้าสู่ระบบ : (Login)`
   ใช้สำหรับตรวจสอบตัวตนของผู้ใช้จากชื่อผู้ใช้หรืออีเมลและรหัสผ่าน จากนั้นสร้าง Session และพาไปยังหน้าตามบทบาท
3. `ระบบกู้คืนรหัสผ่าน : (ForgotPassword, RequestPasswordReset, VerifyPasswordResetOtp, ResetPassword)`
   ใช้สำหรับขอ OTP ยืนยันตัวตน และตั้งรหัสผ่านใหม่เมื่อผู้ใช้ลืมรหัสผ่าน
4. `หน้าแรกของระบบ : (Home)`
   ใช้แสดงสินค้าเด่น โปรโมชันสาธารณะ และข้อมูลเบื้องต้นของร้านในหน้าแรก
5. `หน้าเมนูสินค้า : (Menu)`
   ใช้แสดงรายการสินค้าทั้งหมดที่พร้อมขาย พร้อมข้อมูลราคา หมวดหมู่ และจำนวนคงเหลือ
6. `หน้าโปรโมชัน : (Promotion)`
   ใช้แสดงรายการโปรโมชันที่กำลังเปิดใช้งาน รวมถึงโปรโมชันกิจกรรมที่ผู้ใช้สามารถส่งคำขอรับสิทธิ์ได้
7. `หน้าข้อมูลติดต่อ : (Contact)`
   ใช้เชื่อมไปยังหน้าแรกของระบบ โดยในโปรเจกต์นี้ไม่ได้มีหน้าติดต่อแยกเฉพาะ
8. `ระบบจัดการโปรไฟล์ : (Profile)`
   ใช้สำหรับดูและแก้ไขข้อมูลบัญชีของผู้ใช้ รวมถึงอัปเดตข้อมูลส่วนตัวจากหน้าโปรไฟล์
9. `ระบบจัดการที่อยู่ : (SaveAddress, DeleteAddress, GetUserAddresses)`
   ใช้สำหรับเพิ่ม แก้ไข ลบ และดึงรายการที่อยู่ของผู้ใช้ไปใช้งานในหน้าโปรไฟล์และหน้า Checkout
10. `ระบบสั่งซื้อสินค้า : (Checkout, CreateOrder, GetCurrentUser)`
    ใช้สำหรับเตรียมคำสั่งซื้อ ตรวจสอบข้อมูลผู้ใช้ และบันทึกออเดอร์ลงฐานข้อมูล
11. `ระบบใช้งานโปรโมชันและขอรับโปรโมชัน : (GetUserPromos, SubmitPromotionClaim)`
    ใช้สำหรับดึงโปรโมชันของผู้ใช้ไปใช้ใน Checkout และส่งคำขอรับโปรโมชันกิจกรรมพร้อมหลักฐาน
12. `ระบบชำระเงิน : (Payment)`
    ใช้แสดงยอดชำระเงินและ QR Code สำหรับโอนเงินผ่าน PromptPay
13. `ระบบอัปโหลดสลิป : (UploadSlip)`
    ใช้รับไฟล์หลักฐานการชำระเงิน บันทึกไฟล์ และอัปเดตสถานะการชำระเงินของคำสั่งซื้อ
14. `ระบบติดตามสถานะคำสั่งซื้อ : (Delivery)`
    ใช้แสดงสถานะปัจจุบันของออเดอร์ ข้อมูลการจัดส่ง และประวัติคำสั่งซื้อที่เสร็จสิ้นแล้ว
15. `ระบบยืนยันได้รับสินค้า : (CompleteOrder)`
    ใช้สำหรับให้ผู้ใช้ยืนยันว่าได้รับสินค้าแล้ว และย้ายข้อมูลไปยังประวัติคำสั่งซื้อ
16. `ระบบการแจ้งเตือน : (GetMyNotifications, MarkAsRead, MarkAllAsRead, ClearMyNotifications)`
    ใช้สำหรับดึง อ่าน และล้างการแจ้งเตือนของผู้ใช้

#### ฝั่งพนักงาน (Staff)

1. `ระบบจัดการออเดอร์ : (Order, GetOrderDetails)`
   ใช้สำหรับดูรายการออเดอร์ อ่านรายละเอียดสินค้า และตรวจสอบข้อมูลการสั่งซื้อของลูกค้า
2. `ระบบยืนยันการชำระเงิน : (ConfirmPayment)`
   ใช้สำหรับตรวจสอบหลักฐานการชำระเงิน อัปเดตสถานะการจ่ายเงิน และตัดสต็อกสินค้าตามออเดอร์
3. `ระบบเตรียมสินค้าและจัดส่ง : (AcceptOrder, ShipOrder)`
   ใช้สำหรับรับออเดอร์เข้าสู่ขั้นตอนเตรียมสินค้า และเปลี่ยนสถานะเป็นจัดส่งเมื่อพร้อมส่ง
4. `ระบบจัดการสต็อกสินค้า : (Stock, UpdateStockSettings)`
   ใช้สำหรับดูจำนวนคงเหลือของสินค้า และอัปเดตสต็อกหรือสถานะพร้อมขายของสินค้า
5. `ระบบตรวจสอบคำขอโปรโมชัน : (EventPromotion, ApproveClaim, RejectClaim)`
   ใช้สำหรับตรวจสอบคำขอโปรโมชันจากลูกค้า พร้อมอนุมัติหรือปฏิเสธตามหลักฐานที่ส่งมา

#### ฝั่งผู้ดูแลระบบ (Admin)

1. `ระบบแดชบอร์ด : (Dashbordadmin)`
   ใช้สำหรับดูภาพรวมของระบบ เช่น ข้อมูลยอดขาย คำสั่งซื้อ และข้อมูลสรุปที่สำคัญ
2. `ระบบจัดการสมาชิก : (Member, SaveMember, DeleteMember)`
   ใช้สำหรับดู เพิ่ม แก้ไข และลบข้อมูลผู้ใช้ในระบบ รวมถึงกำหนดบทบาทของผู้ใช้
3. `ระบบจัดการออเดอร์ : (Order, GetOrderDetails, ConfirmPayment, AcceptOrder, ShipOrder)`
   ใช้สำหรับดูรายละเอียดออเดอร์ ตรวจสอบการชำระเงิน และควบคุมสถานะของคำสั่งซื้อ
4. `ระบบจัดการสต็อกและหมวดหมู่ : (Stock, UpdateStockSettings, SaveCategory, SaveStock)`
   ใช้สำหรับเพิ่มหรือแก้ไขสินค้า ปรับจำนวนสต็อก และจัดการหมวดหมู่สินค้า
5. `ระบบจัดการโปรโมชัน : (PromotionAdmin, SavePromotion)`
   ใช้สำหรับสร้าง แก้ไข และบันทึกข้อมูลโปรโมชันของร้านลงในระบบ
6. `ระบบมอบโปรโมชันให้ผู้ใช้ : (GiftPromotion, GiftPromotionToAll)`
   ใช้สำหรับมอบสิทธิ์โปรโมชันให้ผู้ใช้รายบุคคลหรือมอบให้ผู้ใช้ทั้งหมดตามเงื่อนไขที่กำหนด
7. `ระบบตรวจสอบคำขอโปรโมชัน : (EventPromotion, ApproveClaim, RejectClaim)`
   ใช้สำหรับตรวจสอบคำขอโปรโมชันกิจกรรมจากลูกค้า และแจ้งผลการอนุมัติกลับไปยังผู้ใช้

## ตัวอย่างการอธิบาย

## 1. หน้า Login

### 1. ViewModel ที่ใช้
หน้า Login ใช้ `AuthUserViewModel` เป็นตัวกลางในการรับข้อมูลจากฟอร์มเข้าสู่ระบบ โดยในหน้านี้ใช้งานหลักที่ `Username` และ `Password` แม้ว่า ViewModel ตัวนี้จะถูกใช้ร่วมกับหน้า Signup ด้วย แต่ในส่วน Login จะเน้นเฉพาะข้อมูลที่ใช้ตรวจสอบตัวตนของผู้ใช้

รูปที่ควรแปะ

- `Viewmodels/AuthUserViewModel.cs` บรรทัด 1-10

อธิบายใต้รูป

- `AuthUserViewModel` อยู่ในโฟลเดอร์ `Viewmodels`
- หน้า Login ใช้ property `Username` และ `Password`
- `Username` รองรับทั้งชื่อผู้ใช้และอีเมล
- `Password` ใช้เปรียบเทียบกับข้อมูลรหัสผ่านในฐานข้อมูล

### 2. Controller
การทำงานของหน้า Login อยู่ใน `AccountController` action `Login` ทั้งแบบ `GET` และ `POST`

รูปที่ควรแปะ

- `Controllers/AccountController.cs` บรรทัด 21-68
- ถ้าแคปไม่หมดในรูปเดียว ให้แบ่งเป็น 2 รูปต่อเนื่องกัน แต่ยังถือว่าเป็นการอธิบาย Controller ชุดเดียวกัน

ลำดับการทำงาน

1. ระบบเปิดหน้า Login โดยเรียก `GET Login()` เพื่อสร้าง `AuthUserViewModel` และส่งไปยังหน้า View
2. เมื่อผู้ใช้กรอก `Username` และ `Password` แล้วกดเข้าสู่ระบบ ข้อมูลจะถูกส่งกลับมายัง `POST Login(AuthUserViewModel model)`
3. ในส่วน `POST Login(...)` Controller จะตรวจสอบก่อนว่าผู้ใช้กรอกข้อมูลครบหรือไม่ หากกรอกไม่ครบจะเพิ่มข้อความผิดพลาดและส่งกลับไปหน้าเดิม
4. จากนั้นระบบจะค้นหาผู้ใช้จากตาราง `Users` โดยรองรับทั้งการกรอกชื่อผู้ใช้และอีเมล
5. เมื่อพบข้อมูลผู้ใช้แล้ว ระบบจะตรวจสอบว่ารหัสผ่านที่กรอกตรงกับข้อมูลในฐานข้อมูลหรือไม่
6. หากไม่พบผู้ใช้หรือรหัสผ่านไม่ถูกต้อง ระบบจะบันทึก log การเข้าสู่ระบบไม่สำเร็จ พร้อมแสดงข้อความ `Invalid username or password`
7. หากข้อมูลถูกต้อง ระบบจะบันทึกข้อมูลสำคัญของผู้ใช้ลง `Session` เช่น `UserId`, `Username`, `RoleId` และ `RoleName`
8. สุดท้ายระบบจะตรวจสอบสิทธิ์ของผู้ใช้ แล้ว redirect ไปยังหน้าที่เหมาะสมตามบทบาท ได้แก่ Admin ไป Dashboard, Staff ไป Order และ User ไป Home

### 1. ViewModel ที่ใช้

### 2. Controller
การทำงานของระบบกู้คืนรหัสผ่านอยู่ใน `AccountController` โดยแยกเป็น `ForgotPassword`, `RequestPasswordReset`, `VerifyPasswordResetOtp` และ `ResetPassword`

รูปที่ควรแปะ

- ถ้าแคปไม่หมดในรูปเดียว ให้แบ่งเป็นหลายรูปต่อเนื่องกัน แล้วอธิบายรวมตามลำดับการทำงาน

ลำดับการทำงาน

1. ระบบเปิดหน้า `ForgotPassword()` เพื่อแสดงหน้ารีเซ็ตรหัสผ่านให้ผู้ใช้กรอกข้อมูลในหน้าเดียว
2. เมื่อผู้ใช้กรอกอีเมลและกดขอรหัส ระบบจะเรียก `RequestPasswordReset(...)`
3. Controller ตรวจสอบก่อนว่าอีเมลถูกกรอกและมีอยู่จริงในตาราง `Users`
4. ถ้าพบผู้ใช้ ระบบจะสร้าง OTP แบบตัวเลข 4 หลัก พร้อมกำหนดเวลาหมดอายุ 5 นาที แล้วบันทึกลงในตาราง `Users`
5. หลังจากนั้นเมื่อผู้ใช้กรอก OTP ระบบจะเรียก `VerifyPasswordResetOtp(...)` เพื่อตรวจสอบว่าอีเมล รหัส OTP และเวลาหมดอายุยังถูกต้อง
6. ถ้า OTP ถูกต้อง ระบบจะบันทึกสถานะการยืนยันไว้ใน `Session` เพื่อใช้ในขั้นตอนตั้งรหัสผ่านใหม่
7. เมื่อผู้ใช้กรอกรหัสผ่านใหม่และยืนยันรหัสผ่าน ระบบจะเรียก `ResetPassword(...)`
8. Controller จะตรวจสอบอีกครั้งว่า session การยืนยัน OTP ยังถูกต้อง รหัสผ่านใหม่ตรงกัน และมีความยาวอย่างน้อย 6 ตัวอักษร
9. ถ้าทุกเงื่อนไขผ่าน ระบบจะอัปเดตรหัสผ่านใหม่ในตาราง `Users` แล้วล้างค่า OTP และ session ที่เกี่ยวข้องออก
10. สุดท้ายระบบจะส่งผลลัพธ์กลับไปให้หน้าเว็บ เพื่อแจ้งว่าการรีเซ็ตรหัสผ่านสำเร็จ

4. ระบบจะค้นหาสิทธิ์โปรโมชันที่ยังไม่ถูกใช้จาก `UserPromotions` และรวมข้อมูลจาก `Promotions`
5. ถ้าผู้ใช้ต้องการขอรับโปรโมชันกิจกรรม ระบบจะส่งฟอร์มไปยัง `SubmitPromotionClaim(...)`
6. Controller จะตรวจสอบว่าโปรโมชันยังใช้งานได้ รับไฟล์รูปที่ถูกประเภท ขนาดไม่เกินกำหนด และยังไม่มีคำขอค้างอยู่
7. ถ้าผ่านเงื่อนไข ระบบจะบันทึกไฟล์หลักฐานลงในโฟลเดอร์ `wwwroot/uploads/promotion-claims`
8. จากนั้นระบบจะสร้างข้อมูลคำขอใหม่ในตาราง `PromotionClaims` และแจ้งเตือนไปยังผู้ใช้

## 11. ระบบชำระเงิน

### 11.1 ViewModel ที่ใช้
หน้า Payment ไม่ได้ใช้ ViewModel แยกเฉพาะหน้า แต่ Controller ส่ง `Order` ไปยัง View โดยตรง และใช้ `ViewBag` ช่วยส่งข้อมูลยอดเงินกับ `OrderId`

รูปที่ควรแปะ

- `Views/Account/Payment.cshtml` บรรทัด 1-2
- `Views/Account/Payment.cshtml` บรรทัด 174-201

อธิบายใต้รูป

- หน้า Payment ใช้ `@model Order`
- `ViewBag.TotalAmount` และ `ViewBag.OrderId` ใช้แสดงข้อมูลที่จำเป็นสำหรับการชำระเงิน

### 11.2 Controller
การทำงานของระบบชำระเงินอยู่ใน `OrderController.Payment`

รูปที่ควรแปะ

- `Controllers/OrderController.cs` บรรทัด 277-303

ลำดับการทำงาน

1. ผู้ใช้เข้าสู่หน้า `Payment(int orderId)` หลังจากสร้างออเดอร์สำเร็จ
2. Controller ตรวจสอบก่อนว่าผู้ใช้เข้าสู่ระบบอยู่จริง
3. จากนั้นระบบจะค้นหาออเดอร์ของผู้ใช้จากตาราง `Orders` พร้อมข้อมูลรายการสินค้าและโปรโมชัน
4. ถ้าไม่พบออเดอร์ ระบบจะ redirect กลับไปหน้า Home
5. เมื่อพบออเดอร์แล้ว ระบบจะใช้ `PromptPayHelper.GeneratePayload(...)` สร้างข้อมูล payload สำหรับ PromptPay
6. จากนั้นระบบจะสร้าง QR Code ด้วย `QRCoder`
7. Controller ส่งข้อมูล QR Code, `OrderId` และยอดเงินรวมไปยังหน้า View เพื่อให้ผู้ใช้ชำระเงิน

### 11.3 View
หน้า Payment อยู่ใน `Views/Account/Payment.cshtml`

รูปที่ควรแปะ

- `Views/Account/Payment.cshtml` บรรทัด 174-201

อธิบายใต้รูป

- มีส่วนสำหรับรับไฟล์สลิปจากผู้ใช้
- มีส่วนแสดง QR Code สำหรับให้ผู้ใช้ชำระเงิน

### 11.4 ผลลัพธ์ที่ผู้ใช้เห็น

- ผู้ใช้จะเห็นหน้า Payment พร้อมยอดชำระเงินและ QR Code
- ผู้ใช้สามารถใช้ข้อมูลนี้ไปชำระเงินก่อนอัปโหลดสลิป
- หลังจากนั้นจึงเข้าสู่ขั้นตอนยืนยันการชำระเงินผ่านการแนบหลักฐาน

### 11.5 ตารางฐานข้อมูลที่เกี่ยวข้อง

## 12. ระบบอัปโหลดสลิป

### 12.1 ViewModel ที่ใช้
ระบบอัปโหลดสลิปไม่ได้ใช้ ViewModel แยก แต่รับข้อมูลผ่าน parameter โดยตรง คือ `orderId` และ `IFormFile slipImage`

รูปที่ควรแปะ

- `Controllers/OrderController.cs` บรรทัด 307-347

อธิบายใต้รูป

- ระบบใช้ `orderId` เพื่ออ้างอิงว่ากำลังอัปโหลดสลิปให้คำสั่งซื้อใด
- ใช้ `slipImage` รับไฟล์หลักฐานการชำระเงินจากผู้ใช้

### 12.2 Controller
การอัปโหลดสลิปอยู่ใน `OrderController.UploadSlip`

รูปที่ควรแปะ

- `Controllers/OrderController.cs` บรรทัด 307-347

ลำดับการทำงาน

1. หลังจากผู้ใช้ชำระเงินแล้ว หน้า Payment จะส่งไฟล์สลิปไปยัง `UploadSlip(int orderId, IFormFile slipImage)`
2. Controller ตรวจสอบก่อนว่าผู้ใช้ login อยู่จริงและออเดอร์นั้นเป็นของผู้ใช้คนนั้น
3. ระบบตรวจสอบว่าแนบไฟล์เข้ามาจริงหรือไม่
4. จากนั้นระบบจะสร้างชื่อไฟล์ใหม่ แล้วบันทึกไฟล์ไว้ในโฟลเดอร์ `wwwroot/uploads/slips`
5. เมื่อบันทึกไฟล์สำเร็จ ระบบจะอัปเดต `SlipImagePath` และ `PaymentStatus = "PendingVerify"` ในตาราง `Orders`
6. หลังจากนั้นระบบจะสร้าง `Notification` เพื่อแจ้งว่าระบบได้รับสลิปแล้วและกำลังรอการตรวจสอบ
7. สุดท้ายระบบจะส่งผลลัพธ์กลับไปยังหน้าเว็บ เพื่อให้ redirect ไปยังหน้า Delivery
